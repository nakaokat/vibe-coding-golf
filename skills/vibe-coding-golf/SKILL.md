# Vibe Coding Golf

タスクの開始から完了までの会話を分析し、「ストローク数（人間の介入回数）」をゴルフのスコア方式で評価・報告する。

## 目的

「少ない指示でどれだけ高品質なタスクを完了できるか」をゲーム感覚で測定する。

- **理想スコア（ホールインワン）**: 人間のメッセージ1回（初期指示のみ）+ ツール承認0回
- スコアが少ないほど、初期プロンプトの品質とパーミッション設定が最適化されている
- count-interactions スキルの集計に加え、ゴルフ風のスコアリングと称号付与を行う

## ストローク数の集計

### Step 1: ストロークを分類する

以下を分類し、種別ごとの重みでストロークを計算する:

| ストローク種別 | 重み | 説明 | 例 |
|---|---|---|---|
| **初期指示** | 1.0 | タスクの最初の依頼 | 「バグを修正して」 |
| **追加指示・判断** | 1.0 | Claudeの質問への回答、方針決定 | 「Aの方針で進めて」 |
| **確認・質問** | 1.0 | 進捗確認や成果物への質問 | 「完了した？」 |
| **ツール承認（推定）** | 0.2 | allowlist外のツール承認イベント | Bash実行の承認など |

ツール承認はパーミッション設定で削減可能な環境要因のため、人間のメッセージ（指示品質）より重みを低くする。合計ストロークは小数点以下を四捨五入する。

### Step 2: パー（基準ストローク数）を設定する

タスクの複雑さに応じてパーを推定する:

| タスク規模 | パーの目安 | 例 |
|---|---|---|
| 簡単（1ファイル変更） | Par 1 | 変数名のリネーム、typo修正 |
| 普通（数ファイル） | Par 2 | 新機能追加、バグ修正 |
| 複雑（設計・調査含む） | Par 3 | リファクタリング、仕様不明瞭なタスク |
| 大規模（複数システム） | Par 4+ | アーキテクチャ変更、大規模実装 |

### Step 3: スコアを計算してスコアカードを表示する

```
## ⛳ Vibe Coding Golf Score Card

| 項目 | 回数 |
|---|---|
| 初期指示 | X 回 | ×1.0 = X |
| 追加指示・判断 | X 回 | ×1.0 = X |
| 確認・質問 | X 回 | ×1.0 = X |
| ツール承認（推定） | 約 X 回 | ×0.2 = X |
| **合計ストローク** | | **X** |
| **パー** | **X** |
| **スコア** | **X（+N / -N / E）** |

### 称号

🏆 [称号]

### 詳細ログ

1. [初期指示] 「（要約）」
2. [ツール承認] （ツール名）× N回
3. [追加指示] 「（要約）」
...

### 改善提案

- ...
```

### Step 4: 称号を授与する

スコア（合計ストローク - パー）に応じた称号を表示する:

| スコア | 称号 | 説明 |
|---|---|---|
| -3以下 | 🦅🦅 アルバトロス以上 | 伝説のプロンプト職人 |
| -2 | 🦅 イーグル | 卓越した指示の明確さ |
| -1 | 🐦 バーディ | 素晴らしいプロンプト品質 |
| E（0） | ⛳ パー | 標準的な達成 |
| +1 | ボギー | もう少し詳しく書けた |
| +2 | ダブルボギー | 初期指示の改善余地あり |
| +3以上 | ⛔ トリプルボギー以上 | 大幅な改善が必要 |

### Step 5: 改善を提案する

次回同じタスクを理想スコアで完了するための具体的な提案を行う:

- **追加指示・判断が発生した場合**: Claudeが質問した内容を特定し、初期プロンプトに含めるべき情報を提案する。改善版プロンプトの例を示す。
- **ツール承認が多い場合**: 頻出するコマンドパターンを特定し、settings.json の allowlist に追加すべきパーミッションを `/update-config` の形式で提案する。
- **確認・質問が発生した場合**: CLAUDE.md やプロンプトに事前に含められる判断基準を提案する。

### Step 6: X（Twitter）シェアリンクを生成する

スコアカードの末尾に、結果をXにポストできるリンクを表示する。

**タスク名の抽象化ルール:**
具体的なタスク内容はプライベートな情報を含む可能性があるため、以下のように抽象化する:
- 具体的なファイル名・機能名・サービス名は含めない
- 「バグ修正」「新機能追加」「リファクタリング」「設定変更」「スキル更新」などカテゴリレベルに留める
- 技術スタック（React、Python等）は含めてよい

投稿テキストのフォーマット（140文字以内を目安に）:

```
⛳ Vibe Coding Golf Result

Task: （抽象化したタスクカテゴリ）
（称号絵文字） （称号名） — （称号説明） / Par （パー）
💬 指示: X  🔄 追加判断: X  ❓ 確認: X  🔧 承認: X

#VibeCodingGolf #ClaudeCode
```

このテキストをURLエンコードして以下の形式で出力する:

```
### X（Twitter）にシェア

👉 [このスコアをポスト](https://twitter.com/intent/tweet?text=（URLエンコード済みテキスト）)
```

- 日本語・絵文字はすべてパーセントエンコードする
- 改行は `%0A`、スペースは `%20`、`#` は `%23` に変換する
- リンクをクリックするとXがテキスト入力済みの状態で開く

## 注意事項

- このスキルは会話の最後（タスク完了後）に呼び出されることを想定している
- スキル自体の呼び出し（「スコアを教えて」等）はカウント対象に含めない
- ツール承認の推定値には「推定」と明記し、正確性の限界を伝える
- パーはタスクの性質を考慮して適切に設定する（不明な場合は Par 2 をデフォルトとする）
